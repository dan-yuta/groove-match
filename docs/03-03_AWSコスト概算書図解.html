🔍 3. サービス選定理由

各カテゴリごとに候補を比較し、最適なサービスを選定。

3.1 コンピュート: ECS Fargate

採用
ECS Fargate
メリット
  • サーバー管理不要
  • オートスケーリング
  • Docker対応
デメリット
  • EC2比で単価が高い
不採用
ECS on EC2
メリット
  • 単価が安い
  • GPUサポート
デメリット
  • EC2の管理が必要
  • AMI更新が必要
不採用
Lambda + API Gateway
メリット
  • 完全サーバーレス
  • 最低コスト
デメリット
  • Next.js SSRとの相性が悪い
  • コールドスタート
不採用
App Runner
メリット
  • 最もシンプル
デメリット
  • VPC内リソースアクセスが制限的
  • カスタマイズ性が低い
不採用
Lightsail Container
メリット
  • 安価で簡単
デメリット
  • スケーリングの柔軟性が低い
選定理由: Next.js SSRを安定稼働させるには常時起動のコンテナが必要。
Fargateならインスタンス管理不要で、ECSのオートスケーリングもそのまま使える。
Dockerfileベースなのでローカル環境との差異も少ない。

3.2 データベース: Aurora PostgreSQL Serverless v2

採用
Aurora Serverless v2
メリット
  • 自動スケーリング（0.5〜16 ACU）
  • Multi-AZ標準
  • 管理不要
デメリット
  • 最低0.5 ACUの課金が発生
不採用
Aurora Provisioned
メリット
  • 固定性能で予測しやすい
デメリット
  • 手動スケーリングが必要
不採用
RDS PostgreSQL
メリット
  • 安価
  • 実績豊富
デメリット
  • フェイルオーバー手動
  • バージョン管理が必要
不採用
DynamoDB
メリット
  • 完全サーバーレス
  • 超低遅延
デメリット
  • リレーショナルクエリに不向き
  • 設計が複雑
選定理由: Serverless v2ならトラフィックに応じて自動スケール。
深夜帯は最小ACUまで縮小するためコスト効率が良い。
Multi-AZもデフォルトで有効。
バックアップも自動。
マッチングのような複雑なクエリにはリレーショナルDBが最適。

3.3 キャッシュ: ElastiCache Serverless（Redis）

採用
ElastiCache Serverless
メリット
  • ノード管理不要
  • 自動スケーリング
  • パッチ適用自動
デメリット
  • プロビジョンド版より若干高い
不採用
ElastiCache (プロビジョンド)
メリット
  • 安価
  • 性能が予測しやすい
デメリット
  • ノードタイプ選定・スケーリング手動
不採用
DynamoDB DAX
メリット
  • DynamoDBとの統合が良い
デメリット
  • DynamoDB前提、用途が限定的
選定理由: セッション管理とマッチング結果キャッシュ用。
Serverless版ならノードの監視・フェイルオーバー・パッチ適用すべて不要。

3.4 ファイルストレージ: S3 + CloudFront

採用（デファクトスタンダード）
S3 + CloudFront

AWSでの画像・静的アセット配信のデファクトスタンダード。
特に比較不要。
S3にライフサイクルポリシーを設定し、未使用の一時ファイルを30日で自動削除。

Amazon Cognito
メリット
  • マネージド認証
  • ソーシャルログイン標準搭載
  • 5万MAU無料
デメリット
  • UIカスタマイズに制約あり
不採用
自前実装（JWT）
メリット
  • 完全カスタマイズ可能
デメリット
  • セキュリティリスク
  • メンテコスト大
不採用
Auth0
メリット
  • 高機能、UI優秀

Firebase Auth
メリット
  • 簡単、無料枠大きい

5万MAUまで無料なので初期コストゼロ。
Google/Appleログインも標準対応。

3.6 メール: Amazon SES

採用
Amazon SES

月10,000通程度なら月額$1以下。

3.7 CI/CD: GitHub Actions + ECR

採用
GitHub Actions
メリット
  • GitHubに統合済み
  • YAMLベースで簡単
  • 無料枠2,000分/月
デメリット
  • 大規模だと課金が増える
不採用
CodePipeline + CodeBuild
メリット
  • AWS完結
  • IAM統合
デメリット
  • 設定が複雑
  • UIが使いにくい
不採用
CircleCI
メリット
  • 高機能
  • 並列実行
デメリット
  • 外部サービス依存
  • 有料
選定理由: リポジトリがGitHubなので最も自然な選択。
無料枠で十分足りる。
ECRへのプッシュもGitHub公式アクションで簡単。

3.8 監視: CloudWatch + SNS

採用
CloudWatch + SNS

AWS標準の監視ツール。
追加ツール不要でECS/Aurora/ALBのメトリクスを一元管理。
SNSでSlack通知連携。

3.9 IaC: AWS CDK（TypeScript）

採用
CDK (TypeScript)
メリット
  • アプリと同じ言語で記述可能
  • 型安全
  • 抽象度が高い
デメリット
  • 学習コストが若干ある
不採用
CloudFormation (YAML)
メリット
  • AWS標準
  • ドキュメント豊富
デメリット
  • 記述量が多い
  • 可読性が低い
不採用
Terraform
メリット
  • マルチクラウド対応
  • エコシステム充実
デメリット
  • HCL学習が必要
  • 状態管理が複雑
選定理由: Next.jsプロジェクトと同じTypeScriptで記述できるため、学習コストが最小。
L2コンストラクトが高い抽象度を提供し、少ないコードでインフラ構築可能。

⚠ NAT Gateway コストに関する注意

NAT Gatewayは固定費（約$32/月）＋データ処理費がかかり、小規模アプリでは大きな割合を占めます。
