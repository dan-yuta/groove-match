## 2. サービス選定方針：運用しやすさ重視

個人・小規模チームでの運用を想定し、以下の方針でサービスを選定しています。

| 方針 | 詳細 |
|------|------|
| マネージドサービス優先 | サーバー管理・パッチ適用が不要なサービスを選択 |
| サーバーレス活用 | スケーリング自動化、使った分だけ課金 |
| 運用作業の最小化 | バックアップ・監視・ログが組み込まれたサービス |
| シンプルな構成 | 学習コストが低く、トラブルシュートしやすい構成 |

## 3. サービス選定理由

### 3.1 コンピュート: ECS Fargate（選定）

| 候補 | メリット | デメリット | 判定 |
|------|---------|-----------|------|
| **ECS Fargate** | サーバー管理不要、オートスケーリング、Docker対応 | EC2比で単価が高い | **採用** |
| ECS on EC2 | 単価が安い、GPUサポート | EC2の管理が必要、AMI更新が必要 | 不採用 |
| Lambda + API Gateway | 完全サーバーレス、最低コスト | Next.js SSRとの相性が悪い、コールドスタート | 不採用 |
| App Runner | 最もシンプル | VPC内リソースへのアクセスが制限的、カスタマイズ性が低い | 不採用 |
| Lightsail Container | 安価で簡単 | スケーリングの柔軟性が低い | 不採用 |

**選定理由**: Next.js SSRを安定稼働させるには常時起動のコンテナが必要。Fargateならインスタンス管理不要で、ECSのオートスケーリングもそのまま使える。Dockerfileベースなのでローカル環境との差異も少ない。

### 3.2 データベース: Aurora PostgreSQL Serverless v2（選定）

| 候補 | メリット | デメリット | 判定 |
|------|---------|-----------|------|
| **Aurora Serverless v2** | 自動スケーリング（0.5〜16 ACU）、Multi-AZ標準、管理不要 | 最低0.5 ACUの課金が発生 | **採用** |
| Aurora Provisioned | 固定性能で予測しやすい | 手動スケーリングが必要 | 不採用 |
| RDS PostgreSQL | 安価、実績豊富 | フェイルオーバー手動、バージョン管理が必要 | 不採用 |
| DynamoDB | 完全サーバーレス、超低遅延 | リレーショナルクエリに不向き、設計が複雑 | 不採用 |

**選定理由**: Serverless v2ならトラフィックに応じて自動スケール。深夜帯は最小ACUまで縮小するためコスト効率が良い。Multi-AZもデフォルトで有効。バックアップも自動。マッチングのような複雑なクエリにはリレーショナルDBが最適。

### 3.3 キャッシュ: ElastiCache Serverless（Redis）（選定）

| 候補 | メリット | デメリット | 判定 |
|------|---------|-----------|------|
| **ElastiCache Serverless** | ノード管理不要、自動スケーリング、パッチ適用自動 | プロビジョンド版より若干高い | **採用** |
| ElastiCache (プロビジョンド) | 安価、性能が予測しやすい | ノードタイプ選定・スケーリング手動 | 不採用 |
| DynamoDB DAX | DynamoDBとの統合が良い | DynamoDB前提、用途が限定的 | 不採用 |

**選定理由**: セッション管理とマッチング結果キャッシュ用。Serverless版ならノードの監視・フェイルオーバー・パッチ適用すべて不要。

### 3.4 ファイルストレージ: S3 + CloudFront（選定）

特に比較不要。S3 + CloudFrontはAWSでの画像・静的アセット配信のデファクトスタンダード。

**追加**: S3にライフサイクルポリシーを設定し、未使用の一時ファイルを30日で自動削除。

### 3.5 認証: Amazon Cognito（選定）

| 候補 | メリット | デメリット | 判定 |
|------|---------|-----------|------|
| **Cognito** | マネージド認証、ソーシャルログイン標準搭載、5万MAU無料 | UIカスタマイズに制約あり | **採用** |
| 自前実装（JWT） | 完全カスタマイズ可能 | セキュリティリスク、メンテコスト大 | 不採用 |
| Auth0 | 高機能、UI優秀 | 外部SaaS依存、有料プランが高い | 不採用 |
| Firebase Auth | 簡単、無料枠大きい | AWSエコシステムとの統合が弱い | 不採用 |

**選定理由**: AWSサービスとのIAM統合がシームレス。5万MAUまで無料なので初期コストゼロ。Google/Appleログインも標準対応。

### 3.6 メール: Amazon SES（選定）

月10,000通程度なら月額$1以下。Cognitoと統合して認証メールも自動送信。

### 3.7 CI/CD: GitHub Actions + ECR（選定）

| 候補 | メリット | デメリット | 判定 |
|------|---------|-----------|------|
| **GitHub Actions** | GitHubに統合済み、YAMLベースで簡単、無料枠2,000分/月 | 大規模だと課金が増える | **採用** |
| CodePipeline + CodeBuild | AWS完結、IAM統合 | 設定が複雑、UIが使いにくい | 不採用 |
| CircleCI | 高機能、並列実行 | 外部サービス依存、有料 | 不採用 |

**選定理由**: リポジトリがGitHubなので最も自然な選択。無料枠で十分足りる。ECRへのプッシュもGitHub公式アクションで簡単。

### 3.8 監視: CloudWatch + SNS（選定）

AWS標準の監視ツール。追加ツール不要でECS/Aurora/ALBのメトリクスを一元管理。SNSでSlack通知連携。

### 3.9 IaC: AWS CDK（TypeScript）（選定）

| 候補 | メリット | デメリット | 判定 |
|------|---------|-----------|------|
| **CDK (TypeScript)** | アプリと同じ言語で記述可能、型安全、抽象度が高い | 学習コストが若干ある | **採用** |
| CloudFormation (YAML) | AWS標準、ドキュメント豊富 | 記述量が多い、可読性が低い | 不採用 |
| Terraform | マルチクラウド対応、エコシステム充実 | HCL学習が必要、状態管理が複雑 | 不採用 |

**選定理由**: Next.jsプロジェクトと同じTypeScriptで記述できるため、学習コストが最小。L2コンストラクトが高い抽象度を提供し、少ないコードでインフラ構築可能。

## 9. NAT Gateway コストに関する注意

NAT Gatewayは固定費（約$32/月）＋データ処理費がかかり、小規模アプリでは大きな割合を占めます。

### 代替策

| 方法 | メリット | デメリット |
|------|---------|-----------|
| VPCエンドポイント（S3, ECR等） | NAT経由のトラフィック削減 | 全サービスには対応していない |
| NAT Instance（t4g.nano） | 月額約$4で済む | 自前管理が必要、帯域制限 |
| パブリックサブネットにFargate配置 | NAT不要 | セキュリティリスクが増加 |

**推奨**: まずはNAT Gateway 1台 + 主要サービスのVPCエンドポイントで運用開始。拡大期にAZ冗長化。
