⇄ リクエストフロー

ユーザーのリクエストがレスポンスとして返るまでの流れ

1

DNS解決

ユーザーが bandmatch.jp にアクセス → Route 53 が CloudFront のエンドポイントに解決

2

WAFフィルタリング

AWS WAF がリクエストを検査。
ボット・SQLi・XSS・レートリミット超過をブロック

3

CDN判定

CloudFront がキャッシュヒットなら即返却。
ミスなら静的 → S3、動的 → ALB にルーティング

4

ロードバランシング

ALB が正常な ECS Fargate タスクにリクエストを振り分け（ラウンドロビン + ヘルスチェック）

5

アプリ処理

Next.js (SSR) が Cognito でトークン検証 → ElastiCache でキャッシュ確認 → Aurora にクエリ

6

データ取得

RDS Proxy 経由で Aurora にアクセス。
読み取りはリードレプリカに自動ルーティング

7

レスポンス返却

SSR レンダリング結果を ALB → CloudFront → ユーザーに返却。
CloudFront がキャッシュ保存

🛡 セキュリティ多層防御

外側から内側へ5層のセキュリティレイヤーでアプリケーションを保護

Layer 1: エッジ防御

AWS WAF（DDoS / ボット / SQLi / XSS 防御）+ CloudFront（TLS 1.2+ 強制）+ AWS Shield Standard

Layer 2: ネットワーク分離

VPC + パブリック/プライベートサブネット分離 + セキュリティグループ（最小権限ポート開放）+ NACLs

Layer 3: 認証・認可

Cognito（JWT + MFA対応）+ IAMロール（最小権限原則）+ API認可（管理者ロール制限）

Layer 4: アプリケーション

入力バリデーション + CSPヘッダー + CSRF対策 + パラメータバインディング（Prisma ORM）

Layer 5: データ保護

Aurora暗号化（AES-256）+ Secrets Manager（自動ローテーション）+ S3暗号化 + bcryptパスワードハッシュ

🛡
セキュリティグループ フロー:
ALB (443/tcp from 0.0.0.0) → ECS (3000/tcp from ALB-SG) → RDS Proxy (5432/tcp from ECS-SG) → Aurora
ECS (6379/tcp) → ElastiCache | すべてのアウトバウンドはNAT Gateway経由
