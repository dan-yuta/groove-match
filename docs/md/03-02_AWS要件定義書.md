# BandMatch AWS要件定義書

## 1. プロジェクト概要

| 項目 | 内容 |
|------|------|
| プロジェクト名 | BandMatch |
| 概要 | 初心者コピバン向けコミュニティアプリ（Web + iOS + Android 対応）。<br>やりたい曲・アーティストで仲間を見つけてコピバンを結成 |
| ドメイン | bandmatch.jp |
| ターゲット | コピバンをやりたい楽器初心者〜中級者（18〜35歳中心）。<br>同じ曲をやりたい仲間が見つからない人 |
| 収益モデル | サブスクリプション（Free ¥0 / Pro ¥500、初回2週間無料トライアル）<br>※ iOS/Android はストア手数料15%が発生 |

## 3. 非機能要件

### 3.1 性能要件

| 項目 | 要件 |
|------|------|
| レスポンス時間 | ページ表示 2秒以内（P95） |
| API応答時間 | 500ms以内（P95） |
| 同時接続数 | 初期 1,000ユーザー、1年後 10,000ユーザー |
| スループット | 100リクエスト/秒（初期） |

### 3.2 可用性要件

| 項目 | 要件 |
|------|------|
| 稼働率 | 99.9%（月間ダウンタイム約43分以内） |
| 計画停止 | 月1回、深夜帯（2:00-5:00）に実施 |
| 災害対策 | 東京リージョン内のマルチAZ構成 |

### 3.3 セキュリティ要件

| 項目 | 要件 |
|------|------|
| 通信暗号化 | TLS 1.2以上（HTTPS必須） |
| パスワード保存 | bcrypt（ストレッチング回数10以上） |
| 認証トークン | JWT（有効期限24時間、リフレッシュトークン30日） |
| 入力バリデーション | サーバーサイドで全入力検証 |
| XSS対策 | HTMLエスケープ、CSPヘッダー設定 |
| CSRF対策 | CSRFトークン検証 |
| SQLインジェクション対策 | パラメータバインディング（ORMのみ使用） |
| API認可 | 管理APIは管理者ロールのみアクセス可 |
| 個人情報保護 | メールアドレスの暗号化保存、アクセスログの保持（90日） |
| WAF | AWS WAF によるボット対策・レートリミット |

### 3.4 モバイルアプリ要件

| 項目 | 要件 |
|------|------|
| ストアガイドライン | App Store / Google Play のガイドライン準拠 |
| オフライン対応 | オフライン時の基本機能（キャッシュ済みデータ閲覧） |
| アプリ起動時間 | 2秒以内 |

### 3.5 拡張性要件

| 項目 | 要件 |
|------|------|
| 水平スケーリング | ECSタスク数の自動調整 |
| DB接続 | コネクションプーリング（RDS Proxy） |
| CDN | 静的アセットはCloudFront配信 |

### 4.2 AWSサービス一覧

| カテゴリ | サービス | 用途 | 構成 |
|----------|---------|------|------|
| DNS | Route 53 | ドメイン管理・ルーティング | bandmatch.jp ホストゾーン |
| CDN | CloudFront | 静的アセット配信・SSL終端 | カスタムドメイン + ACM証明書 |
| ストレージ | S3 | 静的アセット・画像アップロード | 2バケット（static / uploads） |
| ロードバランサー | ALB | HTTPSリクエスト振り分け | パブリックサブネット、ヘルスチェック |
| コンテナ | ECS Fargate | Next.jsアプリ実行 | 最小2タスク、オートスケーリング |
| コンテナレジストリ | ECR | Dockerイメージ管理 | ライフサイクルポリシー（最新10イメージ保持） |
| データベース | Aurora PostgreSQL | メインDB | db.t4g.medium、Multi-AZ、自動バックアップ7日 |
| DB接続管理 | RDS Proxy | コネクションプーリング | Fargateからの接続最適化 |
| キャッシュ | ElastiCache (Redis) | セッション・マッチング結果キャッシュ | cache.t4g.micro、Single-AZ（初期） |
| メール送信 | SES | パスワードリセット・通知メール | 東京リージョン |
| 認証 | Cognito | ユーザー認証・ソーシャルログイン | ユーザープール + Google/X/Instagram/Apple連携 |
| シークレット管理 | Secrets Manager | DB接続情報・APIキー管理 | 自動ローテーション有効 |
| 監視 | CloudWatch | ログ・メトリクス・アラーム | ECS/RDS/ALBのメトリクス監視 |
| セキュリティ | WAF | 不正リクエスト防止 | CloudFrontに適用、レートリミット |
| IaC | CloudFormation or CDK | インフラのコード管理 | 環境別スタック（dev/staging/prod） |
| プッシュ通知 | SNS | プッシュ通知配信（APNs / FCM） | モバイルアプリ向け通知 |
| ストレージ（モバイル） | S3 | アプリバイナリ配信用（OTAアップデート） | Capacitor Live Update |

### 4.3 ネットワーク構成

```
VPC: 10.0.0.0/16 (ap-northeast-1)
├── Public Subnet AZ-a  : 10.0.1.0/24  → ALB, NAT Gateway
├── Public Subnet AZ-c  : 10.0.2.0/24  → ALB
├── Private Subnet AZ-a : 10.0.10.0/24 → ECS Fargate, ElastiCache
├── Private Subnet AZ-c : 10.0.20.0/24 → ECS Fargate
├── Private Subnet AZ-a : 10.0.30.0/24 → Aurora (Writer)
└── Private Subnet AZ-c : 10.0.40.0/24 → Aurora (Reader)
```

| 項目 | 設定 |
|------|------|
| リージョン | ap-northeast-1（東京） |
| AZ | ap-northeast-1a, ap-northeast-1c |
| NAT Gateway | AZ-a に1台（初期）→ 本番はAZ毎に配置 |
| セキュリティグループ | ALB用、ECS用、RDS用、Redis用を分離 |

## 6. CI/CD パイプライン

```
GitHub (main)
    │
    ▼ push / PR マージ
GitHub Actions
    │
    ├── ① lint + 型チェック + テスト
    │
    ├── ② Docker ビルド → ECR プッシュ
    │
    ├── ③ staging デプロイ（ECS サービス更新）
    │
    ├── ④ E2Eテスト（staging環境で実行）
    │
    ├── ⑤ 本番デプロイ（手動承認後）
    │
    └── ⑥ モバイルビルド
        ├── build:mobile → Capacitor sync
        ├── Xcode Cloud（iOS） → App Store
        └── Fastlane（Android） → Google Play
```

| ステップ | ツール | 詳細 |
|----------|--------|------|
| コード品質 | ESLint + TypeScript | PR時に自動実行 |
| テスト | Jest + React Testing Library | ユニットテスト・コンポーネントテスト |
| E2Eテスト | Playwright | 主要フロー（登録→マッチング→バンド作成）の自動テスト |
| ビルド | Docker | マルチステージビルド（ビルド層 + 実行層） |
| デプロイ | ECS Rolling Update | ダウンタイムゼロ |
| 本番デプロイ | 手動承認 | GitHub Actions の手動トリガー |
