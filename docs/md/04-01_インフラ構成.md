## 2. IaC: CloudFormation

### CDKからCloudFormationへの変更理由

| 項目 | 説明 |
|------|------|
| **Before** | AWS CDK (TypeScript) |
| **After** | **CloudFormation**（YAML/JSONテンプレート） |
| **理由** | CloudFormationはAWSの標準IaCツール。CDKは内部的にCloudFormationに変換されるため、直接CloudFormationを使うことでデバッグが容易。GitHub ActionsからのStack更新も直接的 |

### デプロイフロー

```
Git push → GitHub Actions → CloudFormation Stack更新
```

1. 開発者がコードをGitHubにpush
2. GitHub Actionsが自動で起動
3. CloudFormationテンプレートを使ってAWSリソースを作成・更新

## 3. 3環境構成

Groove Matchでは、**dev（開発）・staging（ステージング）・prod（本番）**の3環境を用意します。

| 環境 | 用途 | URL | 構成 |
|------|------|-----|------|
| dev | 開発・検証 | dev.groovematch.jp | 最小構成（ECS 1タスク、RDS最小） |
| staging | リリース前テスト | staging.groovematch.jp | 本番と同構成（縮小版） |
| prod | 本番 | groovematch.jp | フル構成 |

> **なぜ3環境?**: 開発中のバグが本番に影響しないよう、段階的にテストしてからリリースするためです。stagingは本番と同じ構成なので、本番リリース前に「本番と同じ条件でテスト」できます。

```
                 +---------+ +---+----------+----+
                                 |          |

                       |      Aurora に移行          |
                       +----------------------------+
```

### 用語説明

| サービス | 説明 |
|---------|------|
| Route 53 | AWSのDNSサービス。ドメイン名（groovematch.jp）とサーバーのIPアドレスを紐づける |
| CloudFront | CDN（Content Delivery Network）。世界中の拠点からコンテンツを高速配信する。SSL（暗号化通信）も担当 |
| S3 | オブジェクトストレージ。画像やCSSなどの静的ファイルを保存・配信する |
| ALB | Application Load Balancer。複数のサーバー（タスク）にリクエストを均等に振り分ける |
| ECS Fargate | サーバーレスコンテナ実行環境。Dockerコンテナ（アプリを動かすための箱）を管理不要で実行できる |
| RDS Proxy | DBへの接続を効率的に管理するプロキシ（中継役）。接続数を最適化する |
| RDS PostgreSQL | AWSのマネージドデータベース。PostgreSQLエンジンで動作 |

## 6. ネットワーク構成 (VPC)

VPC（Virtual Private Cloud）は、AWS上に構築する仮想的なネットワーク空間です。サブネット（ネットワークの区画）を分けて、セキュリティを確保します。

```
VPC: 10.0.0.0/16 (ap-northeast-1)
|-- Public Subnet AZ-a  : 10.0.1.0/24  → ALB, NAT Gateway
|-- Public Subnet AZ-c  : 10.0.2.0/24  → ALB
|-- Private Subnet AZ-a : 10.0.10.0/24 → ECS Fargate, ElastiCache
|-- Private Subnet AZ-c : 10.0.20.0/24 → ECS Fargate
|-- Private Subnet AZ-a : 10.0.30.0/24 → RDS (Writer)
+-- Private Subnet AZ-c : 10.0.40.0/24 → RDS (Reader / 将来追加)
```

| 項目 | 設定 | 説明 |
|------|------|------|
| リージョン | ap-northeast-1（東京） | 日本のユーザー向けなので東京リージョン |
| AZ（Availability Zone） | ap-northeast-1a, 1c | 2つの物理的に離れたデータセンターに分散配置 |
| NAT Gateway | AZ-a に1台（初期） | プライベートサブネットからインターネットにアクセスするための中継装置 |
| セキュリティグループ | ALB用、ECS用、RDS用、Redis用を分離 | 各サービスへのアクセスを必要最小限に制限 |

> **Public Subnet と Private Subnet の違い**: Public Subnetはインターネットから直接アクセスできる区画、Private Subnetはアクセスできない区画です。DBやアプリサーバーをPrivate Subnetに置くことで、外部からの直接攻撃を防ぎます。

## 7. CI/CD パイプライン

CI/CD（Continuous Integration / Continuous Delivery）は、コードの変更を自動でテスト・デプロイする仕組みです。

```
GitHub (main)
    |
    v push / PR マージ
GitHub Actions
    |
    |-- (1) lint + 型チェック + テスト
    |
    |-- (2) Docker ビルド → ECR プッシュ
    |
    |-- (3) CloudFormation Stack 更新
    |
    |-- (4) staging デプロイ（ECS サービス更新）
    |
    |-- (5) E2Eテスト（staging環境で実行）
    |
    |-- (6) 本番デプロイ（手動承認後）
    |
    +-- (7) モバイルビルド
        |-- Xcode Cloud（iOS） → App Store
        +-- Fastlane（Android） → Google Play
```

| ステップ | ツール | 詳細 |
|----------|--------|------|
| コード品質 | ESLint + TypeScript | PR時に自動実行 |
| テスト | Vitest + Playwright | ユニットテスト・E2Eテスト |
| ビルド | Docker | マルチステージビルド（ビルド層 + 実行層） |
| IaC | CloudFormation | テンプレートからAWSリソースを更新 |
| デプロイ | ECS Rolling Update | ダウンタイムゼロ（無停止デプロイ） |
| 本番デプロイ | 手動承認 | GitHub Actions の手動トリガー |

## 8. 非機能要件

### 性能要件

| 項目 | 要件 | 説明 |
|------|------|------|
| レスポンス時間 | ページ表示 2秒以内（P95） | 95%のリクエストが2秒以内に完了 |

| 項目 | 要件 |
|------|------|
| 稼働率 | 99.9%（月間ダウンタイム約43分以内） |

| 災害対策 | 東京リージョン内のマルチAZ構成（2つのデータセンターに分散） |
