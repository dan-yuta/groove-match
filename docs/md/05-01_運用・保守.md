# Groove Match - 運用・保守

## 1. バックアップ

データを失わないための仕組みです。複数の方法を組み合わせて安全性を確保します。

### DB バックアップ

| 方法 | 頻度 | 保持期間 | 説明 |
|------|------|---------|------|
| RDS 自動バックアップ | 毎日（自動） | **7日間** | RDSの標準機能。自動でスナップショット（DBの状態を丸ごとコピーしたもの）を取得 |
| 手動スナップショット | **週1回** | **30日間** | 週末に手動で取得。リリース前にも取得推奨 |

> **スナップショット**とは: データベースのある時点の状態を丸ごと保存したものです。何か問題が起きたときに、その時点の状態に戻すことができます。

### S3 バックアップ

| 設定 | 説明 |
|------|------|
| **バージョニング有効** | ファイルを上書きしても、過去のバージョンが保持される。誤って削除しても復元可能 |
| **ライフサイクルポリシー** | 古いバージョンを一定期間後に自動削除してコストを抑える |

### IaC（インフラコード）バックアップ

CloudFormationテンプレートはGitで管理されるため、**全履歴が保持**されます。過去の任意の時点のインフラ構成に戻すことが可能です。

## 2. 無停止デプロイ（ローリングデプロイ）

ユーザーがアプリを使っている最中でも、サービスを停止せずに新しいバージョンをデプロイする仕組みです。

### 仕組み: ECS Fargate + ALB のローリングデプロイ

```
デプロイ前:
  ALB → [タスクA (旧バージョン)] [タスクB (旧バージョン)]

デプロイ中:
  ALB → [タスクA (旧バージョン)] [タスクC (新バージョン)]
         ※ タスクCが正常起動を確認してからタスクBを停止

  ALB → [タスクD (新バージョン)] [タスクC (新バージョン)]
         ※ タスクDが正常起動を確認してからタスクAを停止

デプロイ完了:
  ALB → [タスクC (新バージョン)] [タスクD (新バージョン)]
```

**ポイント**:
- 古いタスクと新しいタスクが一時的に共存する
- ALB（ロードバランサー）のヘルスチェック（正常動作の確認）に合格してから、古いタスクを停止
- ユーザーから見るとサービスが途切れない

### DB変更時の注意

DB変更（テーブルの追加・カラムの変更等）は**Prismaマイグレーション**で行い、**後方互換性を維持**します。

| 用語 | 説明 |
|------|------|
| マイグレーション | DBの構造（テーブルやカラム）を安全に変更する仕組み |
| 後方互換性 | 新しい変更を加えても、古いバージョンのアプリが動き続けること |

**例**: カラムを追加するときは、まず「NULLを許可するカラム」として追加 → 新バージョンをデプロイ → 古いバージョンが完全に停止してから「NOT NULL制約」を追加

## 3. 監視・アラート

AWSの監視サービス「CloudWatch」を使い、異常を自動検出して通知します。

| 監視対象 | メトリクス | 閾値 | 通知先 | 説明 |
|----------|-----------|------|--------|------|
| ECS | CPU使用率 | > 80% が5分継続 | Slack + メール | サーバーの処理能力が限界に近い |
| ECS | メモリ使用率 | > 80% が5分継続 | Slack + メール | メモリ（RAM）が不足しつつある |
| ALB | 5xxエラー率 | > 1% が3分継続 | Slack + メール | サーバー側エラーが頻発している |
| ALB | レスポンスタイム | P95 > 2秒 が5分継続 | Slack | ページ表示が遅くなっている |
| RDS | CPU使用率 | > 70% が10分継続 | Slack + メール | DBの処理負荷が高い |
| RDS | 空きストレージ | < 10GB | メール | DBの容量が不足しつつある |
| RDS | コネクション数 | > 80% | Slack | DB接続数が上限に近い |
| CloudFront | エラー率 | > 5% が5分継続 | Slack + メール | CDN配信にエラーが発生 |

> **P95**とは: 95パーセンタイル。100回のリクエストのうち95番目に遅いリクエストの時間。「ほぼ全てのユーザーが体感する最悪のレスポンス時間」の目安です。

## 4. 月次メンテナンス

毎月定期的に実施する保守作業です。計画停止は月1回、深夜帯（2:00-5:00）に実施します。

### DBメンテナンス

| 作業 | 説明 | 頻度 |
|------|------|------|
| VACUUM ANALYZE | PostgreSQLの不要データ領域を回収し、クエリの実行計画を最適化する | 月1回 |
| インデックス再構築 | データの検索を高速にするためのインデックス（索引）を再構築 | 月1回 |
| 不要データ削除 | 古い通知データ、期限切れセッション等を削除 | 月1回 |

> **VACUUM ANALYZE** は PostgreSQL 固有のメンテナンスコマンドです。データを削除・更新した後に残る「不要な領域」を回収し、DBを最適な状態に保ちます。

### セキュリティ

| 作業 | 説明 | 頻度 |
|------|------|------|
| OS/ランタイムパッチ適用 | Fargateのコンテナイメージを最新のセキュリティパッチに更新 | 月1回 |
| 依存パッケージ更新 | npm packageの脆弱性を確認し、必要に応じてアップデート | 月1回 |

### モニタリング

| 作業 | 説明 | 頻度 |
|------|------|------|
| コスト確認 | AWS Cost Explorerで前月のコストをレビュー | 月1回 |
| パフォーマンスレビュー | CloudWatchメトリクスでレスポンスタイム・エラー率の傾向を確認 | 月1回 |
| ログ確認 | エラーログの傾向分析、不審なアクセスの確認 | 月1回 |

### バックアップ検証

| 作業 | 説明 | 頻度 |
|------|------|------|
| リストアテスト | バックアップからDBを復元できることを実際に確認する | 月1回 |

> **リストアテスト**が重要な理由: バックアップが「正常に取得できている」と「実際に復元できる」は別の話です。いざという時に復元できないことを防ぐため、定期的にテストします。

## 5. 同時アクセス数の目安

Groove Matchがどのくらいの同時アクセスに耐えられるかの目安です。

| 構成 | 同時接続数 | 説明 |
|------|-----------|------|
| Fargate 1タスク | 50〜100 | 最小構成。dev環境やローンチ直後 |
| Fargate 2タスク + ALB | 200〜300 | prod初期構成。2つのタスクで冗長性も確保 |
| Fargate 4タスク + Redis | 500〜1,000 | 成長期。Redisキャッシュでスケジューリングを効率化 |
| + CloudFront | さらに上 | API以外のリクエスト（画像・CSS・JS）はCDNが吸収するため、サーバーの負荷を大幅軽減 |

> **同時接続数**とは: 同じ瞬間にアプリを使っているユーザーの数です。MAU（月間アクティブユーザー）の5〜10%が同時に使うと想定します。

### スケーリングの流れ

```
ローンチ直後（MAU 500）:
  Fargate 2タスク → 同時100接続程度で十分

成長期（MAU 5,000）:
  Fargate 2〜4タスク（オートスケーリング）+ Redis → 同時300〜1,000接続

拡大期（MAU 20,000）:
  Fargate 4〜6タスク + Redis + CloudFront → 同時1,000接続以上
```

**オートスケーリング**: ECS Fargateは、CPU使用率などのメトリクスに基づいて自動でタスク数を増減させることができます。急にアクセスが増えても自動で対応可能です。

## 6. インシデント対応

### 障害レベル定義

| レベル | 条件 | 対応時間 | 例 |
|--------|------|---------|-----|
| Critical | サービス全体停止 | 30分以内 | DB障害、全タスク停止 |
| High | 主要機能が利用不可 | 1時間以内 | マッチング機能停止、決済エラー |
| Medium | 一部機能に影響 | 4時間以内 | 画像アップロード失敗、通知遅延 |
| Low | ユーザー影響なし | 24時間以内 | ログ収集遅延、監視メトリクス欠損 |

### 障害発生時のフロー

```
1. CloudWatch アラーム発火
   |
   v
2. Slack / メールで通知
   |
   v
3. 障害レベル判定
   |
   v
4. 対応（ロールバック / スケールアップ / 修正デプロイ）
   |
   v
5. 復旧確認
   |
   v
6. ポストモーテム（振り返り）作成
```

> **ポストモーテム**とは: 障害が起きた後に「何が起きたか」「なぜ起きたか」「次に同じことが起きないようにするには」をまとめるドキュメントです。
