# Groove Match - 安全対策・リスク管理

> 本ドキュメントは、ユーザー間トラブルの防止策、コミュニケーション設計、無断キャンセル対策、運営の免責方針を定義します。
> 他のマッチングアプリ・コミュニティサービスの事例を調査し、Groove Matchに適した安全設計を行っています。

アプリ全体の機能については [01-02_機能一覧.md](./01-02_機能一覧.md) を参照してください。

---

## 1. 運営の責任範囲と免責方針

### 基本方針

Groove Matchは**「音楽活動のマッチングプラットフォーム」**です。ユーザー同士を繋ぐ「場」を提供し、安全に利用するためのツール（通報・ブロック等）を整備します。**ユーザー間の個別交流・行動について運営は仲介・保証しません。**

### 責任範囲の明確化

| 項目 | 運営が責任を負う | 運営が責任を負わない |
|------|-----------------|-------------------|
| プラットフォーム | 安全なツールの提供（通報・ブロック機能） | ユーザー間の金銭トラブル |
| 通報対応 | 通報への合理的な期間内での対応 | 外部ツール（LINE等）でのやり取り |
| 利用規約 | 禁止行為の明示と違反者への対応 | ライブ出演に関するトラブル（出演キャンセル等） |
| 情報提供 | 安全な利用のための注意喚起 | ユーザーが提供した情報の真偽 |
| 個人情報 | プライバシーポリシーに基づくデータ管理 | ユーザーが自発的に公開した情報の悪用 |

> **免責条項の限界**: 免責条項は「重大な過失」を免除しません。例えば、犯罪行為の通報を放置した場合は過失に該当する可能性があります。通報には必ず合理的な期間内に対応する必要があります。

### 他サービスの免責事例

| サービス | 業種 | 免責方針 |
|---------|------|---------|
| Tinder | マッチング | 「ユーザー間の行為について責任を負わない」を利用規約に明記。ただし通報対応の体制は整備 |
| Bumble | マッチング | ユーザーの身元を保証しない旨を明示。写真認証は努力義務として提供 |
| Pairs | マッチング | 「当社はユーザー間のトラブルに関与しない」が基本方針。ただし24時間有人パトロールを実施 |
| メンボネット | バンド募集 | 宣伝・恋愛目的のアカウントを積極的にフィルタリング。メンバー間トラブルは関与せず |
| Meetup | イベント | グループ運営者に自治権を委譲。プラットフォームはツール提供に徹する |

### 利用規約への同意フロー

ユーザー登録時に**クリックラップ方式**で同意を取得します。

> **クリックラップ方式**とは: チェックボックス + 「同意する」ボタンを用いた同意取得方法です。裁判でも法的拘束力が認められる方式です（スクロールラップ = 最後までスクロールしないとボタンが押せない方式よりは弱いが、十分な法的効力があります）。

```
登録フロー:
  1. 基本情報入力
  2. 音楽プロフィール入力
     ↓
  3. 利用規約・コミュニティガイドラインの表示
     □ 利用規約に同意します
     □ コミュニティガイドラインに同意します
     [同意して登録する]
     ↓
  4. クレジットカード登録
  5. 登録完了
```

### 利用規約で定める禁止行為

| カテゴリ | 禁止行為 | 例 |
|---------|---------|-----|
| 詐欺・金銭 | 金銭の要求・貸し借り | 「機材を一緒に買おう」「スタジオ代を立て替えて」 |
| 勧誘 | MLM・ネットワークビジネスへの勧誘 | 「音楽で稼げるビジネスがある」 |
| 恋愛目的 | 音楽活動以外の出会い目的での利用 | マッチングを恋愛目的で使う |
| なりすまし | 虚偽のプロフィール | 架空のスキルレベル、他人の写真 |
| ハラスメント | 誹謗中傷・脅迫・ストーキング | 執拗なメッセージ、個人情報の暴露 |
| スパム・宣伝 | 営利目的の宣伝行為 | バンドのチケット営業、有料レッスンの営業 |
| 違法行為 | 法律に違反する一切の行為 | 薬物取引、著作権侵害 |

### 利用規約とコミュニティガイドラインの違い

| 項目 | 利用規約 | コミュニティガイドライン |
|------|---------|----------------------|
| 法的拘束力 | あり（契約） | なし（推奨ルール） |
| 違反時の措置 | アカウント停止・BAN | 注意喚起・警告 |
| 内容 | 禁止行為、免責事項、個人情報の取扱い | マナー、推奨行動、コミュニケーション指針 |
| 更新手続き | 変更時に再同意が必要 | 通知のみ |

## 2. ユーザー安全対策

### 2.1 想定リスク一覧

| リスク種別 | 具体例 | 深刻度 | 対策 |
|-----------|--------|--------|------|
| 詐欺・金銭トラブル | 機材購入詐欺、スタジオ代持ち逃げ、偽のブッキング | Critical | 通報機能、注意喚起、金銭の要求を検出 |
| MLM/ネットワークビジネス | 音楽活動を口実にした勧誘、「稼げるビジネス」への誘導 | High | 禁止ワードフィルタ、通報、利用規約で禁止 |
| 犯罪行為 | ストーカー、ハラスメント、暴力 | Critical | ブロック機能、通報→即時対応、警察への情報提供体制 |
| なりすまし | 偽プロフィール、架空のスキルレベル、他人の写真 | High | プロフィール審査（将来）、通報機能 |
| 宣伝・スパム | バンドのチケット営業、有料レッスンの勧誘 | Medium | 禁止ワードフィルタ、送信レート制限 |
| 恋愛目的の利用 | マッチングを出会い系として利用 | Medium | 利用規約で禁止、通報機能 |

### 2.2 音楽マッチング特有のリスク

Groove Matchは出会い系アプリではありませんが、**見知らぬ人と実際に会う**（スタジオ練習、ライブ出演）点で独自のリスクがあります。

| リスク | 通常のマッチングアプリとの違い | 対策 |
|--------|---------------------------|------|
| 対面リスク | 「スタジオで会おう」は正常な使い方 | 初回練習時の安全アドバイス表示 |
| 金銭のやり取り | スタジオ代・ライブ出演費の割り勘は正常 | 「アプリ外での金銭のやり取りは自己責任」を明示 |
| 個人情報の共有 | LINEグループへの参加は正常なバンド活動 | 外部連絡先の交換時に注意喚起を1回表示 |

#### 音楽業界特有の詐欺パターン

| パターン | 手口 | 警告サイン |
|---------|------|-----------|
| 偽ブッキング | 「大きなイベントに出演できる」と持ちかけ、出演費を前払いさせる | 前払いの要求、実在しないイベント会場 |
| 偽レーベル/マネジメント | 「デビューさせる」と誘い、登録料や写真撮影代を要求 | 前払いの要求、正式な契約書がない |
| 機材詐欺 | 「安く譲る」と持ちかけ、代金だけ受け取って逃げる | アプリ外での個人間売買 |
| MLM勧誘 | 「音楽で稼げる」とセミナーや教材を売りつける | 「参加費」「ビジネスモデル」「不労所得」等のキーワード |

### 2.3 多層防御アーキテクチャ

4つのレイヤーで安全対策を実施します。

```
┌──────────────────────────────────┐
│    レイヤー1: 予防                 │
│    登録時の規約同意               │
│    プロフィール審査（将来対応）    │
│    初回対面時の安全アドバイス      │
├──────────────────────────────────┤
│    レイヤー2: 検出                 │
│    通報機能                       │
│    禁止ワードフィルタ（Pro）       │
│    送信レート制限                 │
├──────────────────────────────────┤
│    レイヤー3: 対応                 │
│    警告 → 機能制限 → 停止 → BAN  │
│    通報対応SLA                    │
├──────────────────────────────────┤
│    レイヤー4: 事後                 │
│    ブロック機能                   │
│    マッチング結果からの除外       │
│    必要に応じて警察への情報提供   │

```

#### 他サービスの多層防御事例

| サービス | 予防 | 検出 | 対応 | 参考にした点 |
|---------|------|------|------|-------------|
| Tinder | AI不正検出、写真認証 | レッドフラグ言語の自動スキャン | 通報→48時間以内の対応 | エスカレーション段階制 |
| Bumble | Deception Detector™ | AIプロフィールスキャン | 匿名通報、警告→永久BAN | 自動検出の考え方 |

| メンボネット | 宣伝アカウントの手動フィルタ | ユーザー通報 | 該当アカウント削除 | 音楽特化の禁止行為 |

### 2.4 通報・ブロック機能

#### 通報カテゴリ

| カテゴリ | 説明 | 対応優先度 |
|---------|------|-----------|
| 犯罪行為 | ストーカー、脅迫、暴力等 | Critical（24時間以内） |
| 詐欺・金銭トラブル | 金銭の騙し取り、持ち逃げ | High（48時間以内） |
| 勧誘・ネットワークビジネス | MLM、宗教勧誘等 | High（48時間以内） |
| ハラスメント | 誹謗中傷、セクハラ、差別的発言 | High（48時間以内） |
| なりすまし | 虚偽のプロフィール、他人の写真 | Medium（72時間以内） |
| スパム・宣伝 | 営利目的の宣伝、大量メッセージ | Medium（72時間以内） |
| マナー違反 | コミュニティガイドラインへの軽微な違反 | Low（1週間以内） |

#### 通報フロー

```
ユーザーが通報ボタンをタップ
  ↓
通報カテゴリを選択
  ↓
詳細説明を入力（任意）
  ↓
通報キューに追加（通報者は匿名）
  ↓
運営が通報をレビュー
  ├─ 事実確認
  ├─ 過去の通報履歴確認
  └─ 対象ユーザーのメッセージ履歴確認
  ↓
対応を実行
  ├─ 警告メール送信
  ├─ 機能制限（メッセージ停止等）
  ├─ アカウント一時停止
  └─ 永久BAN
  ↓
通報者に結果を通知（「対応しました」のみ。具体的な制裁内容は通知しない）
```

#### ブロック機能

ブロックしたユーザーは以下から**完全に除外**されます:

- マッチング検索結果
- フレンド候補
- コピバンメンバー候補
- コミュニティのタイムライン
- イベント参加者リスト上での表示

ブロックされた側には**ブロックされたことは通知されません**。

### 2.5 エスカレーションポリシー

| 段階 | トリガー | アクション | 復帰条件 |
|------|---------|-----------|---------|
| 1回目 | 初回の通報で違反確認 | 警告メール送信 | 即時復帰 |
| 2回目 | 2回目の違反確認 | メッセージ送信停止（7日間） | 7日後に自動復帰 |
| 3回目 | 3回目の違反確認 | アカウント一時停止（30日間） | 30日後に運営審査の上で復帰 |
| 4回目 or Critical | 4回目の違反またはCritical通報 | **永久BAN** | 復帰不可 |

> **Critical通報**（犯罪行為等）の場合は、通報回数に関係なく即座に永久BANの対象となります。

### 2.6 異性紹介事業との関係

> **インターネット異性紹介事業**とは: 異性との出会いを目的としたサービスを提供する事業で、都道府県公安委員会への届出が必要です（出会い系サイト規制法）。

Groove Matchは**音楽活動のマッチングが目的**であり、異性紹介事業には該当しない可能性が高いと考えられます。ただし以下の対策で明確化します:

- 利用規約で**恋愛目的の利用を明確に禁止**
- マッチングは**楽器・曲・ジャンル・スキル・スケジュール**ベース（性別ベースではない）
- 恋愛目的の利用が通報された場合は利用規約違反として対応

> **注意**: これは法的助言ではありません。正式な判断は弁護士に相談することを推奨します。

## 3. コミュニケーション設計

### 3.1 課題

コピバン活動には連絡が不可欠です:

- 練習日時の調整
- スタジオの予約連絡
- セットリストの相談
- イベント出演の打ち合わせ

この連絡を**アプリ内で完結**させるか、**外部ツール（LINE等）に委ねる**かを決定する必要があります。

### 3.2 比較: アプリ内メッセージ vs 外部ツール

| 観点 | アプリ内メッセージ | 外部ツール（LINE等） |
|------|------------------|-------------------|
| **安全性** | ◎ 運営が監視可能、不審パターンを検出できる | △ 運営は関与できない |
| **プライバシー** | △ E2E暗号化なし（運営がメッセージを確認可能） | ◎ LINEはE2E暗号化対応 |
| **利便性** | △ グループ通話・日程調整・ファイル共有は未対応 | ◎ 既に慣れたツール、豊富な機能 |
| **運営の責任** | 範囲内（監視義務が生じる可能性） | 範囲外（明確に免責可能） |
| **実装コスト** | 高い（リアルタイムチャット基盤が必要） | なし |
| **不正対策** | ◎ 禁止ワードフィルタ、通報機能との連携 | × 対策不可能 |
| **課金動機** | ◎ Pro限定機能として課金転換に貢献 | × 課金動機にならない |

### 3.3 推奨: ハイブリッド方式（3フェーズモデル）

Pairs等の実績を参考に、**段階的なコミュニケーション設計**を採用します。

```
フェーズ1: アプリ内メッセージ（初回接触）
  ├─ Pro限定機能
  ├─ 運営が監視可能
  ├─ 不審パターンの検出
  └─ Free→Pro転換の課金動機
  ↓
フェーズ2: 外部連絡先の交換（信頼構築後）
  ├─ ユーザーの自主判断
  ├─ 注意喚起を1回表示
  │   「外部での連絡は運営の監視対象外です。
  │    個人情報の共有は慎重に判断してください。」
  └─ 運営は外部でのやり取りに関与しない
  ↓
フェーズ3: 外部ツールでのバンド活動（LINEグループ等）
  ├─ 練習日時の調整
  ├─ グループ通話
  ├─ ファイル共有
  └─ 完全に運営の責任範囲外
```

#### なぜハイブリッドなのか

- **アプリ内のみ**: バンド活動の実務（グループ通話、日程調整等）にはLINEの方が圧倒的に便利。アプリ内に閉じ込めるとユーザー体験が悪化し、離脱につながる
- **外部のみ**: 初対面の相手にいきなりLINEを教えるのは不安。初回接触時の安全網がなくなる
- **ハイブリッド**: 初回はアプリ内で安全に接触 → 信頼関係ができたら自然にLINEへ移行。実務上最も現実的

### 3.4 アプリ内メッセージの安全機能

| 機能 | 優先度 | 説明 |
|------|--------|------|
| メッセージ通報 | 必須 | 個別メッセージから直接通報できる導線 |
| 禁止ワードフィルタ | 高 | 勧誘・詐欺の典型パターンを検出（「参加費」「ビジネスモデル」「不労所得」等） |
| URL送信警告 | 高 | 外部URLを含むメッセージに注意喚起を表示 |
| 送信レート制限 | 中 | スパム防止（1分あたり最大10件） |
| AI不正検出 | 低（将来） | MLモデルによる不審会話の自動検出 |

### 3.5 SNS連携の安全設計

既存のSNS連携（X / Instagram / LINE / YouTube）はプロフィール表示用ですが、安全面の考慮が必要です。

| リスク | 対策 |
|--------|------|
| SNS経由でのストーキング | SNS連携はオプトイン（ユーザーが自発的に設定） |
| 個人情報の特定 | フレンド承認済みユーザーのみにSNSリンクを表示（検討項目） |
| プロフィールと実際のSNSアカウントの不一致 | 連携はOAuth認証で検証 |

## 4. 無断キャンセル・ゴースティング対策

### 4.1 想定シナリオ

| シナリオ | 影響 | 深刻度 |
|---------|------|--------|
| ライブ出演バンドがドタキャン | イベント運営に多大な迷惑。他バンドのタイムテーブルにも影響 | Critical |
| メンバーがライブ当日にバックレ | バンド全体が出演不可能になる可能性。他メンバーの練習が無駄に | Critical |
| メンバーが練習に来ない | バンド活動が停滞。他メンバーのモチベーション低下 | High |
| マッチング後に無反応（ゴースト） | 相手の時間を浪費。アプリへの不信感 | Medium |
| フレンド承認後にゴースト | 影響は限定的 | Low |

### 4.2 基本方針

**運営はユーザー間の人間関係に介入しません。** バンド内のトラブル（ドタキャン、ゴースト等）は個人の問題であり、運営が解決すべきものではありません。

ただし、**トラブルを予防・可視化する仕組み**は提供します:

- レピュテーション（信頼度）スコアで実績を可視化
- キャンセルポリシーの明示で期待値を統一
- バンドリーダーへの自治権付与

### 4.3 他サービスの事例

| サービス | 対策 | 参考にした点 |
|---------|------|-------------|
| Meetup | グループ運営者がノーショーへの罰金設定、キャンセル窓の設定、常習者の除名権限 | グループ自治の考え方 |
| Airbnb | ホスト/ゲスト双方の評価制度、スーパーホストステータス | 双方向のレピュテーション |
| Uber | ドライバー/乗客の相互評価、低評価が続くとアカウント停止 | スコアによる品質維持 |

### 4.4 レピュテーション（信頼度）スコア

自動ペナルティは導入しません。代わりに**ソフトなレピュテーション制**を採用します。

> **レピュテーションスコア**とは: ユーザーの活動実績に基づく信頼度の数値です。プロフィールに★マーク（1〜5段階）で表示され、他のユーザーから見えます。

#### スコアに影響する行動

| 行動 | スコア影響 | 説明 |
|------|-----------|------|
| ライブ出演完了 | +10 | イベントへの参加実績 |
| 練習セッション参加 | +3 | バンドメンバーからの確認 |
| バンドメンバーからの高評価 | +5 | バンド活動後の相互評価 |
| 期限内キャンセル（3日前以上） | 0 | ペナルティなし |
| 直前キャンセル（24時間以内） | -5 | 注意表示 |
| 無断キャンセル（ノーショー） | -15 | プロフィールに影響 |

**スコアの計算ロジックは非公開**とします（ゲーミング = スコアを不正に操作する行為を防止するため）。

#### スコアの表示

| スコア範囲 | 表示 | 意味 |
|-----------|------|------|
| 80-100 | ★★★★★ | 非常に信頼できる |
| 60-79 | ★★★★☆ | 信頼できる |
| 40-59 | ★★★☆☆ | 普通 |
| 20-39 | ★★☆☆☆ | 注意が必要 |
| 0-19 | ★☆☆☆☆ | 要注意 |

新規ユーザーは**スコア50（★★★☆☆）**からスタートします。

### 4.5 キャンセルポリシー

イベント出演に関するキャンセルポリシーです。**登録時の利用規約同意に含まれます。**

| 期限 | 対応 | レピュテーション影響 |
|------|------|-------------------|
| 7日前以上 | ペナルティなし | なし |
| 3〜7日前 | 内部記録（ユーザーには表示しない） | なし |
| 24時間〜3日前 | バンドリーダーに通知 | 小（-5） |
| 24時間以内 | バンドリーダー + イベント主催者に通知 | 大（-10） |
| 無断キャンセル（ノーショー） | 自動警告メール + 記録 | 大（-15） |

### 4.6 バンド内の自治機能

バンド内の人間関係は**バンドリーダーの自治**に委ねます。

| 権限 | 説明 |
|------|------|
| メンバー除外 | バンドリーダーがメンバーをバンドから除外できる |
| メンバー評価 | バンド活動後に相互評価（レピュテーションスコアに反映） |
| 募集の開閉 | メンバー募集中フラグの切り替え |
| バンド内参加率の表示 | バンド内でのみ見える、各メンバーの練習参加率 |

運営はバンド内のトラブルに介入しません。バンドリーダーが解決する責任を持ちます。

| 金銭の貸し借りはしないでください | 金銭トラブルの防止 |
| 不安を感じたらすぐにその場を離れてください | 身の安全を最優先 |
